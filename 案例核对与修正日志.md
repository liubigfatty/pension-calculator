## 操作规则（大哥确认）
1. **外部标尺**：以大哥提供的**真实社保局核定表**为唯一权威，不以现有 cases 册当真理。
2. **三项分比**：每个案例分别比对 `基础养老金 / 个人账户养老金 / 过渡性养老金`，定位差异段落。
3. **逐步修正**：一个省 / 一个案例核对清楚再动下一个，不批量瞎改。
4. **留记录**：每步写进本文件 + 当日 memory 日志。
5. **可回滚**：每次实质修改前确保 git 有基线快照；改坏了用 `git revert <commit>` 或 `git checkout -- <file>` 回到基线。

## 方法论备忘（为什么不能信 cases 册）
- cases 册 31 省均不可全信：含早期引擎输出回填值（git 1269a37）、零覆盖（无中人→过渡=0 未测）、逻辑矛盾（sight=0 却过渡>0）。
- **"案例通过" ≠ "引擎正确"**：通过只代表 引擎 == expected，而 expected 可能脏。
- 只有真实核定表（四项金额官方打印）能当尺子。
- **预发机制**：退休当年基数未发布前制表→用上一年基数预发（多退少补）。表上"上年度在岗平均工资"=退休前一年基数，常<引擎当年基数。遇此优先判预发，用表上基数喂引擎核对，勿当社平bug（详见 MEMORY「预发机制」段）。

## 测试脚手架根因修复（2026-07-10 下午，重大）
- **发现两个测试根因，导致此前"15省通过/16省失败"的结论全部作废**：
  1. `scripts/run-cases.js` 直接把省份 `module.exports` 喂给 `calculate()`，**绕过了 `getEngineConfig()`**。而生产（云函数 index.js）走 `provModule.getEngineConfig()`。结果：测试路径下省份模块参数（过渡系数 TRANS_COEF、特殊增发）取不到→过渡算成 NaN；生产路径却正常。测试 ≠ 生产。
  2. `run-cases.js` 对比时把 `null/NaN` 实际值当成"通过"（`Math.abs(NaN - exp) > 1` 为 false）→ 引擎没算出过渡的案例被**静默判通过**。
- **修复**：① run-cases.js 改用 `getEngineConfig()`（与 production 一致）；② 实际值为 null/NaN 时判失败。两处修复后测试才真实反映生产。
- **诚实基线（修复后全量）**：173 案例（吉林新增 case5），100 通过，73 失败；吉林、新疆已结清。73 个失败仍集中在 `transitional_pension` 字段。失败分两类：
  - A 类「脏案例」：expected.transitional=0 但引擎算出真实值（如 chongqing/2、3，与 xinjiang/3 同类）→ 把 transitional 字段补正即可。
  - B 类「真公式/系数偏差」：引擎值与核定值差较大（如 chongqing/1：1514 vs 1837）→ 需逐省用真实核定表校准引擎（系数/视同指数/公式）。
- **纠正旧结论**：引擎过渡在生产里一直能算（系数在 getEngineConfig 内部设了），并非"22省 NaN"。之前"16省失败"是测试假象 + cases 脏基准叠加。cases 全量修正真正剩余工作 = 47 个过渡值校准（A类补字段、B类对引擎）。
- xinjiang.js 额外修复：导出 TRANS_COEF（让 getEngineConfig 之外的消费方也能取到），并在 getEngineConfig 启用冬季采暖补贴120元/月（新政办发[2010]167号），使 total 与官方核定表一致。
