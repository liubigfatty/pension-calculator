# 社平数据审计与修复报告（2026-07-10）

## 一、官网核对你的框架："发布年 = 使用年"
✅ **框架正确**。官方口径原文（社保网/搜狐）："2025年使用的就是2024年的平均工资数据"——文件命名里的年份 = 它生效使用的年份；底层统计数据滞后一年是数据细节，不影响存储口径。

落地到引擎：`AVG_SALARY_HISTORY[Y]`（社平）与 `PROV_BASE[Y]`（计发基数）都按**使用年 Y** 存储，二者不是"计发基数 = 上年社平"的换算关系。

> ⚠️ 期间我一度误判"整体 AVG=PROV_BASE 同年对齐"能修全部问题，实测发现这是**错的**——详见第三节。

## 二、引擎计算逻辑审计
✅ **引擎逻辑本身正确，无需改代码**（Explore 子代理审计 + 主程序回归测试双重确认）。

- `pension-engine.js` 中 `AVG_SALARY_HISTORY` 仅用于**个人账户余额估算**（同年读取，无年份偏移）。
- 基础养老金 / 过渡性养老金 / 指数化月平均缴费工资 都用 `PROV_BASE`（同年），与 AVG 无关。
- **缴费指数是输入项**，引擎不重算（index-mini 才用 AVG 算指数）。
- 唯一 `-1` 偏移在 `:1525`（重庆"用上年度社平"）——**法定要求，必须保留**。
- 回归测试：`scripts/_test_ai_pension.js` **22/22 通过（含 31 省冒烟）**；缴费指数引擎 `index-engine/test-calcIndex.js` 收敛完成。

## 三、关键修正：AVG ≠ PROV 不是 bug
全量扫描 31 省发现 **28 省 `AVG ≠ PROV` 同年值**，但其中绝大多数**是故意的、正确的**：

- `AVG_SALARY_HISTORY` = 带小数的真实全口径社平（如浙江 7436.67）。
- `PROV_BASE` = 四舍五入的计发基数（如浙江 7437）。
- 二者真实 **diverge**（吉林 2021 社平 6384.83 vs 计发基数 6004.75，差 6%），并非仅四舍五入。
- **AVG 更精确，对缴费指数计算反而更好** → 这 22 个省一律**保留不动**。

唯一无辩驳的真 bug 是"**相邻年份社平完全相等**"（真社平不可能精确相等）。
