## 第 3 章　数据修改 SOP（标准作业流程 + 多端同步操作链）

> 核心认知：**改真相源只是第一步，漏跑同步链会导致"网页/小程序/AI 显示旧数据"**。下面是从改数据到全端生效的完整链路。

1. **备份先行**：改 `provinces/` 前先 `cp -r provinces backup_provinces_YYYYMMDD/`（放在 `cloudfunctions/calculate/` 下，事后可删）。
2. **精确替换**：在 `PROV_BASE { }` 块内**按年份精确替换值**，保留原有注释与文号（勿重建整个块，否则丢失标定文号注释）。
3. **年份口径**：`PROV_BASE` 的 key = 使用年 / 退休年（黑龙江例外，见 4.2）。**官方未公布的年份（如 2026）不写任何值**，交给引擎外推。
4. **同步镜像**：改完 `.js` 必须同步同构 `.json`（通过 `getEngineConfig()` 暴露的字段）。
5. **跑同步脚本（关键，易漏）**：
   - `node scripts/sync-provinces.js` → 把真相源派发到 `data/provinces/`、`js/provinces/`、`docs/...` 等派生副本。
   - `node scripts/build-web.js` → 重新生成 `web/engine.js` + `web/provinces-bundle.js`（网页端）。
6. **重生成 index-mini 社平**（主程序改了 `avg_salary_history` 时必做）：重新生成 `index-mini/cloudfunctions/calcIndex/provinces-data.js`（由主程序 `cloudfunctions/calculate/provinces/*.js` 的 `avg_salary_history` 自动生成，注释标生成时间），再部署 index-mini。
7. **（若在 ai-mode）同步 AI 副本**：把 provinces 复制到 `miniprogram/ai/pension-calc-skill/engine/provinces/` 与 `index-mini/ai/pension-index-skill/engine/`。
8. **跑验证**：`node scripts/run-cases.js` 应 **162/162 全绿**。
9. **专项验证 PROV_BASE**（关键，见第 7 章）：故意不注入 `base_number` 跑案例，确认线上真正读取的 `PROV_BASE` 正确。

> 注：步骤 5–7 的脚本若在 Winclaw 工作区（junction 路径）下用 Write/Edit 落盘失败，改用 Bash + `dangerouslyDisableSandbox` + `node fs` 写文件（见第 8 章陷阱）。

## 第 4 章　数据模型与口径（最易错区）

### 4.1 省份文件顶层结构（真相源 `.js`）

每个 `cloudfunctions/calculate/provinces/<省>.js` 导出一个 `getEngineConfig()`，内含：

| 常量 | 语义 | key 含义 |
|---|---|---|
| `PROV_BASE` | 全省历年计发基数（元/月） | **使用年 / 退休年**（多数省） |
| `AVG_SALARY_HISTORY` | 全省历年全口径社平工资（元/月） | **社平年 / 统计年**（工资所属年） |
| `CC_BASE`（如吉林） | 省直/重点城市独立计发基数（如长春） | 同 PROV_BASE 的退休年 |
| `CITY_BASE` / `SY_BASE` / `DL_BASE` / `SHENZHEN_BASE` / `ZHENGZHOU_BASE` | 城市级独立计发基数 | 同退休年 |
| `BASE_PARAMS` | 计发基数预测参数（含 `PROV_2025` 等占位，注意冲突，见 4.7） | — |
| `TRANS_COEF` | 过渡系数（数字 / `.get()` 函数 / `base+alt`）；可带 `formula_type` | — |
| `MODULES` | 启用的计算模块数组：`['base','extra','personal','transition',...]` | — |
| `_definitions`（全部 31 省 .js 已补写，2026-07-12） | 字段概念/key 含义/口径说明注释块 | 防止社平年/计发年错位 |

> 注意：黑龙江 `.js` 顶部注释仍写着 `// data/provinces/heilongjiang.js`（旧路径，已归档），属**陈旧注释**，不要据此把旧路径当真相源。

### 4.2 两套字典的语义（核心等式）

| 字典 | key 含义 | 存的值 |
|---|---|---|
| `PROV_BASE[Y]` | 使用年 / 退休年（多数省） | Y 年的**计发基数**（元/月） |
| `AVG_SALARY_HISTORY[Y]` | 社平年 / 统计年 | Y 年度的**官方社平工资**（元/月） |

核心等式：**某年计发基数 = 上一年社平工资**。
例：2024 社平 → 2025 计发基数；2025 社平(7705) → 2026 计发基数 / 2026 缴费基数。

> **黑龙江特例（不能套北京式"使用年"）**：黑龙江 `PROV_BASE` 是按**社平年**索引的（与北京/上海等"使用年"索引不同）：
> ```
> PROV_BASE[2024] = 7570   // 2024社平 = 2025计发基数（黑人社函〔2024〕548号「2025年度使用」）
> PROV_BASE[2025] = 7705   // 2025社平 = 2026计发基数/缴费基数（黑人社函〔2025〕611号「2026年度使用」）
> AVG_SALARY_HISTORY[2025] = 7705  // 2025社平
> ```
> 两套不是同一年、不是同一口径，切勿混为一谈。吉林同样存在 AVG≠PROV 同年（吉林社平6384.83 vs 计发基数6004.75 非 bug）。

### 4.3 三级基数（prov / cc / city）

引擎 `getBase()` 支持省、市、城市三级查表，预发回退优先级：**城市表优先于全省表**（`lastCityYear >= lastProvYear` 时取城市值）。

| 级别 | 字段 | 典型省份 |
|---|---|---|
| 全省（prov） | `PROV_BASE` | 全部 |
| 省直/重点市（cc） | `CC_BASE` | 吉林（长春） |
| 城市独立（city） | `SY_BASE`/`DL_BASE`（辽宁沈阳/大连）、`SHENZHEN_BASE`（广东深圳）、`ZHENGZHOU_BASE`（河南郑州） | 辽宁/广东/河南 |

> **城市独立基数范围（2026-07-12 用户决定）**：仅辽宁(沈阳/大连)、广东(深圳)、河南(郑州及洛阳/信阳/开封)建有城市独立计发基数。**菏泽、焦作、农垦暂不建，统一执行省平**——图片中出现的 7506/6385/2223.42 系旧口径或社平口径，不作为独立计发基数。决定已写入 shandong.js / henan.js / heilongjiang.js 注释。

### 4.4 引擎外推规则（getBase，pension-engine.js:1135）

查询年份 `year` 与数据的关系：

1. **精确匹配**：`cityRates[year]` 或 `provRates[year]` 存在 → 直接返回。
2. **查询年 > 数据最大年**：
   - `year == 最大年+1`（**预发年**，当年基数尚未公布）→ 返回**上年原值 flat（不上浮）**。
   - `year > 最大年+1`（远期）→ 按 `GROWTH_RATE`（默认 2%，或 `config.growth_rate`）复利外推。
3. **查询年 < 数据范围**：回退到最早已知值。
4. **区间内缺失年**：从最近已知年按 `GROWTH_RATE` 向后外推。

> 个人账户计算走 `getSalaryBase(..., 'avg_salary_history')` → `getBase(..., 'avg_salary_history')` 经外推（**非直读字典**）。因此**未发布年份不写固定值**，靠引擎运行时外推产生，文件里仅留纯注释占位。

### 4.5 双基数模型（河南等）

- 基础养老金用**全省** `PROV_BASE`（如 6738）。
- 过渡性养老金 / 补贴用**本地** `ZHENGZHOU_BASE`（郑州 2025=7933 等）。
- 引擎 `formula_type='henan'`：基础用 `provBase`、过渡用 `retireBase`（退休地本地基数）。原 `{...PROV_BASE}` 继承全省值会导致郑州过渡性养老金少算约 18%，须显式建 `ZHENGZHOU_BASE`。
- 跨市（洛阳 6573 / 信阳 6260 / 开封 6385）走城市独立基数。

### 4.6 预发机制（最高优先级）

- 退休当年基数未公布 → 用**上年基数预发**。
- 遇到"表上基数 < 当年基数（差约一年）"优先判为**预发**，**绝不默认改引擎或搬数据**。
- 城市有独立计发基数（如湖北）时，预发回退优先使用退休城市基数。

### 4.7 数据口径红线

- `AVG_SALARY_HISTORY` = 元/月，`Y` 槽位存 `Y` 年度官方社平；`PROV_BASE` 公布年即作用年。
- 真实表年月 `32.09` = 32 年 9 个月 = **32.75 年**（不是 32.09 年）。
- 未发布年份**不写任何固定数字**，交引擎外推（杜绝"瞎写估算值"，见历史教训）。
- 注意 `BASE_PARAMS` 里可能残留 `PROV_2025:6738` 之类的误放键（如河南），与 `PROV_BASE` 冲突且会被 `Object.keys().map(Number)` 读成 NaN，清理时一并删除。

## 第 5 章　引擎算法实现（核心计算逻辑）

> 改算法前必读。引擎入口 `calculate(config, inputData)`（pension-engine.js:1436）：先 `parseInput` → 由 `MODULES`/`TRANS_COEF` 装配计算模块 → 依次算 基础/增发/个人账户/过渡/特殊加发。

### 5.1 两条数据线（关键架构）

- **计发基数线**：`getBase(city, year, config, 'base_rates')` → 读 `PROV_BASE`/`CC_BASE`/`CITY_BASE`，用于**基础养老金 + 过渡性养老金**。
- **社平工资线**：`getSalaryBase(city, year, config)` → 内部 `getBase(..., 'avg_salary_history')`，用于**缴费指数 + 个人账户养老金**。

同一个数（如 7010）可能出现在两张表但 key 不同（`prov[2024]=7010` vs `avg[2023]=7010`），概念不同。

### 5.2 基础养老金公式（calcBasicPension，:94）

| 分支 | 公式 | 适用 |
|---|---|---|
| 默认（含 shenzhen） | `(退休地基数 + 退休地基数×指数)/2 × 年限 × 1%` | 绝大多数省、深圳 |
| `henan` | 基础用全省基数 `provBase`：`(provBase + provBase×指数)/2 × 年限 × 1%` | 河南（双基数，基础用全省） |
| `guangdong` | 含 a 系数：`(退休地基数×a + 退休地基数×指数)/2 × 年限 × 1%`，`指数≥0.6 时 a=1`，`<0.6 时 a=指数/0.6`（提升低指数人员待遇，国发[2005]38号） | 广东 |
| `chongqing` 注释 | `(社平 + 社平×Q)/2 × 年限 × 1%`，社平由 `params.socialAvgBase` 传入 | 重庆（代码段见 :115） |

> **已修复的引擎 bug**（:121-125 改）：默认公式应为 `(retireBase + provBase×指数)/2`（国发[2005]38号：指数化工资=全省基数×指数），原代码误用 `retireBase` 两处。对无独立市县基数的省（retireBase=provBase）无影响；对长春(cc)等独立基数省，base 计算偏高。修复后长春案例差从 83.32→0.00。

### 5.3 过渡性养老金（calcTransitionalPension，:395）

按 `formula_type` / `TRANS_COEF.formula_type` 分省实现，差异巨大：

- `beijing`（:411）、`shanghai`（:432）、`jiangsu`（:453）、`sichuan`（:497）、`henan`（:515）、`hubei`（:526）、`shenzhen`（:540）、`chongqing`（:563）、`guangdong`（:572）各有独立过渡公式。
- **湖北关键规则**：过渡性养老金使用**退休地**计发基数（非全省/省直），`formula_type='hubei'`；平均缴费指数按退休地市社平计算；过渡系数 1.2%。吉林则仍用全省基数（两省不可一刀切）。
- `index_floor_one`：指数 < 1 时按 1 处理（内蒙古、宁夏等用，如内蒙古过渡=指数化月均工资×视同年限×1.2%，指数<1按1）。
- `usePreAccountYears=true`（宁夏、云南等）：过渡/视同计算用"建账前年限"而非全部工作年限。

### 5.4 个人账户养老金（calcPersonalAccountPension，:302）

- 记账利率 `getAccRate(year, config)`（:1252）按年取。
- 缴费基数历史 `getBase(city, y, config, 'avg_salary_history')`（:351）经社平线外推，**非直读字典**。

### 5.5 增发 / 特殊加发

- `calcExtraPension`（:155）：四川增发（川劳社发〔2006〕17号）、浙江基本养老金补贴（浙人社发〔2011〕146号，150元/月固定）等。
- `calcSpecialAddition`（:656）：西藏高原补贴（指数化工资×比例，在藏≥10<15→5%/≥15<20→10%/≥20→15%）、青海调节金（2006=150 逐年-15 至2016归零）、云南独生子女（全省人均养老金×5%）等。

## 第 7 章　验证体系（162/162 全绿 ≠ 数据正确）

### 7.1 run-cases 的盲点（重要）

`scripts/run-cases.js` 用案例自带的 `base_number` **注入**引擎，会**绕过 `PROV_BASE`**。所以 162/162 全绿只证明"案例 base + 引擎算法"对，**不验证 `PROV_BASE` 本身**。

但线上 `index.js` **不传 `base_number`**，引擎必从 `PROV_BASE[退休年]` 查基数。因此：

> **改数据必须改 `PROV_BASE` 并验证**；只靠 run-cases 全绿会掩盖 `PROV_BASE` 错误（历史上真实发生过：黑龙江 2023/2024 错位、2026 缺失导致线上取到 undefined/0）。

### 7.2 验证 PROV_BASE 正确性的正确姿势

用真实引擎跑案例时**故意不注入 `base_number`**，强制引擎从 `PROV_BASE[退休年]` 查基数，对比手算：

```
基础养老金 = 计发基数 × (1 + 平均缴费指数) / 2 × 缴费年限 × 1%
```

逐省核对 `PROV_BASE[退休年]` 与官方公布值一致，才算数据验证通过。

### 7.3 验证脚本清单

| 脚本 | 用途 |
|---|---|
| `scripts/run-cases.js` | 全量案例回归（162/162 基准） |
| `scripts/verify-cases.js` / `verify-cases-batch*.js` | 端到端验证（支持不注入 base 强制查 PROV_BASE） |
| `scripts/verify_verified.js` | 已标定(`verified:true`)案例专项验证（注入官方 base_number + 官方参数，对比优先级 `notes手算值 > expected字段`） |
| `scripts/verify-index-mini.js` | index-mini 社平同步验证 |
| `scripts/build-web.js` 后 `scripts/smoke-web.js` | 网页端 build 冒烟验证 |

> 早期验证脚本有三处"假错"根因（字段提取错、未注入官方 base_number 导致预发差、JSON expected 字段录入错），现在正确方法是注入官方 `base_number`+参数、对比 `notes` 手算值。

## 第 8 章　工程规则与陷阱

- **junction 路径**：Winclaw 工作区可能是 junction，node 脚本直接用 `process.cwd()` 拼接路径在 `readdir` 会 ENOENT。`sync-provinces.js`/`build-web.js` 已用 `fs.realpathSync(process.cwd())` 解析。
- **Write/Edit 持久化失败**：在 junction/符号链接下，Write/Edit 可能落不到真实工作区。解法：用 **Bash + `dangerouslyDisableSandbox` + node `fs`** 改文件。
- **同步脚本易漏**：改真相源后必须跑 `sync-provinces.js` + `build-web.js`（见第 3 章）。漏跑→网页/小程序显示旧数据。
- **AI 副本易漏**：在 ai-mode 分支，AI 技能用的 provinces 副本需手动同步（1.4 节）。
- **cases `expected` 不可当官方真值**：早期 `verify --update` 回填过脏值；真实表驱动 > 经验法则。案例库里 `*_generated.json` 脚手架的 expected 多不自洽（total=NaN），属测试噪音，不计入。
- **行业统筹企业**：电力/石油等实际统账时间可能晚于全省统一建账时间，会显著增加视同缴费年限/过渡性养老金。小程序必须支持用户传入实际 `account_start` 年月。
- **双`//`注释无害但略乱**：估算标记 + 原注释叠加，可后续清理但不影响计算。

## 第 10 章　关键教训（防再犯）

1. **路径错选**：多副本 + 无唯一真相源 → 改在过期副本白做（最典型 `WorkBuddy/pension-engine/`）。铁律：所有数据/案例修改只在 `养老金计算平台/` 真实项目做。
2. **口径混用**："上一年社平"和"当年发布的数据"常指数值相同、只是描述角度不同。跟用户沟通时**用其规则原话（"退休年份 N 查 N 对应的基数"）**，别自作主张换"上一年社平"措辞，极易被当成改规则。
3. **计发基数 vs 社平工资**：同一数可能出现在两张表但 key 不同（prov[2024]=7010 vs avg[2023]=7010），概念不同。黑龙江/吉林 AVG≠PROV 同年**不是 bug**。
4. **cases expected 脏值**：早期 verify --update 回填过脏值，不能当官方真值；以真实核定表 + notes 手算值为准。
5. **多城市过渡性养老金**：不能默认用全省基数。湖北表上明确基础/过渡均用退休地基数（荆州 7210），吉林则仍用全省基数。用省份级 `formula_type` 显式区分，避免一刀切改动影响其他省份。
6. **未发布年份瞎写**：曾把 17 省 `AVG[2025]` 套 ~4% 乱涨写成固定值 → 用户纠正"未发布不能写死，靠引擎外推"。改为纯注释占位。
7. **验证全绿≠数据对**：run-cases 注入 base_number 绕过 PROV_BASE；验证 PROV_BASE 必须故意不注入。
