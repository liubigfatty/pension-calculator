# 缴费指数小程序 · 真实数据测试方案

> 数据源：`个人缴费信息查询.xls`（295 条逐月缴费明细）
> 本方案用**最细颗粒度的真实数据**驱动统一表单的几种输入模式，给出可照填的测试用例与预期值，并揭示不同模式下的精度差异。

> 注：EXCEL 中「养老保险缴费基数」为当月基数，「缴费划账户」= 基数×8%（个人部分）。早期基数仅 353 元，后期达 18533 元，说明该职工缴费基数随社平增长而大幅上升。

## 二、金标准（最细颗粒度：逐年明细正向）

把 295 条逐月记录按年聚合成 **25 年逐年明细**，正向计算（指数 = 年基数 ÷ 上年社平，余额按统一记账利率复利）：

- **平均指数 = 1.7836**
- **估算余额（含息） = 204822.85 元**

这是信息最完整、最准的基准值，作为其他模式的对照锚点。

## 四、分析与结论

1. **最细模式（逐年明细）最准**：指数 1.7836，作为基准。适用于能导出逐年/逐月缴费基数的人。
2. **余额反推（填含息余额）接近基准**：指数 1.9888，偏差仅 +11.5%。反推假设"每年指数恒定"求等效常数，故略高于真实加权平均，属正常数学特性。适合"只知道账户余额"的用户。
3. **当前基数法（按现在工资估算）**：指数 0.929，偏差 -47.9%。
   - **原理**：你填的「当前月缴费基数」÷ **当前社平** = 当下指数，把它当作平均缴费指数，反推往年每年基数（= 当年社平 × 当下指数），再算账户余额（含息）。即"用现在的比例，代表过去所有年份"。
   - **本例为何偏低**：该职工真实缴费是"前低后高"（早期基数 353、后期 18533）。当前基数法假设全程按"现在这个比例（0.93）"缴，而真实加权 1.78 是被后期 2.85 的高比例拉高的。所以"只填现在工资"会**低估**历史高比例年份的贡献。
   - 这是"只知道现在工资"时最合理的估算；要想更准，请用「逐年明细」或「账户余额」。
   - ⚠️ 注意：这里填的应是你**现在的**月工资基数，不是历史平均值。本例用 EXCEL 的整段月均 6182.51 仅作演示，真实使用时请填你当前工资对应的缴费基数。
4. **本金和（不含息）偏低**：指数 1.4224，偏差 -20.3%，因未含历年记账利息。填余额时应填社保 App 查到的"个人账户累计存储额"（含息），不是各年本金之和。

## 六、照填验证步骤

**验证金标准（推荐先做）**
1. 参保地选「吉林」
2. 参加时间：1998年07月；累计月数：295
3. 先在基础信息填好「参加工作时间 1998年07月」「累计缴费月数 295」，再点开「逐年明细」——系统自动生成 26 行（年份和月数已填好，从 1998 到 2023）。你只需按上方表格在每行填「月均基数」：1998填353、1999填353……2022填18532.95（有数据的年份填，末行 2023年1月 无数据留空）。**不用写 JSON**。
4. 点计算 → 预期：平均指数 **1.7836**，估算余额 **204822.85**

**验证余额反推**
1. 参保地/参加时间/月数同上
2. 不展开逐年明细；账户余额填 **204822.85**（金标准含息余额，或用你社保 App 真实查到的累计存储额）
3. 点计算 → 预期：反推平均指数 ≈ **1.9888**（与金标准偏差 < 15% 即正常）

**验证当前基数模式（按现在工资估算）**
1. 参保地/参加时间/月数同上
2. 当前月缴费基数填 **6182.51**（即你现在的月工资基数；本例用整段平均值演示）
3. 点计算 → 预期：平均指数 ≈ **0.929**，估算余额 ≈ **95674.26**（这是"假设全程按当前比例"的估算值，与金标准不同属正常，见四.3）

**验证拦截（无输入）**
1. 参保地/参加时间/月数填好，基数和余额都留空
2. 点计算 → 预期：提示"请至少填写月均基数或账户余额之一"
