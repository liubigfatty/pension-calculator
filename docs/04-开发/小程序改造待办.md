## ① 天津双指数（引擎 + 前端，最高优先）

**现状**
- 引擎 `engine/pension-engine.js` 只读取 `inputData.avgIndex / transIndex`，**不自动计算双指数**；cases 里是写死注入 `transIndex` 才过，真实用户输入会退化为单指数。
- 前端 `miniprogram/pages/step2/step2.js:41`：
  ```js
  const DOUBLE_INDEX_PROVINCES = [10, 18, 26, 15]  // 浙江=10, 广东=18, 陕西=26, 河南=15
  ```
  **天津（provinceIndex=1）不在内** → 真实用户没有"过渡性养老金指数（全部平均工资指数）"输入框，断缴人员过渡性养老金算错。
- Web 版 `app.js:43` / `web/app.js:43`：`var DOUBLE_INDEX = { zhejiang: 1, guangdong: 1, shaanxi: 1 }`，**连河南都漏了**。

**影响（四张真实核定表铁证：案例7/8/9/10）**
- 津劳局〔2006〕329号：平均工资指数（基础养老金用，÷应缴年限 n）与全部平均工资指数（过渡性用，分子加视同年×1.00、分母加 S）。
- 断缴场景分母用应缴年限不减 → 两个指数拉开。案例7女无断缴两指数相等(0.82)；案例8/9/10断缴者全部指数(0.81/0.81/1.14)≠平均指数(0.78/0.79/1.17)，方向性100%验证。

**计划改造**
1. 引擎：按津劳局〔2006〕329号**自动算**双指数（平均÷应缴年限 n；全部分子加视同年×1.00、分母加 S；断缴分母不减）。
2. 前端 `DOUBLE_INDEX_PROVINCES` 加天津（索引 1），开"过渡性养老金指数"输入；并补"断缴月数 / 应缴-实缴年限差"输入（普通用户不知"全部指数"是什么）。
3. Web 版 `DOUBLE_INDEX` 同步补齐（浙江、广东、陕西、河南、天津）。

## ② 深圳机关事业 vs 企业计发基数

**现状**
- `guangdong.js`：`city === 'sz'` 一律用 `SHENZHEN_BASE`（2025=11293，企业职工），**未区分机关事业单位**。
- 深圳人社局 2025-11-17 官网问答明确：**机关事业单位按省同期计发基数执行**（2025=9493，不享深圳单列11293）。

**影响**：深圳机关事业人员会被错算成11293（偏高约19%）。现有案例全是企业，未触雷，但真实机关事业用户输入会错。

**计划改造**：引擎按人员类别（企业/机关事业）选基数——企业走 `SHENZHEN_BASE`，机关事业走 `PROV_BASE`（全省）。

## ③ 四川 `predictBase(2026)` 返回 0（低优先）

**现状**：`sichuan.js` `predictBase()` 中 `if (year <= lastYear) return PROV_BASE[year] || 0`，而 `PROV_BASE` 缺 2026 键 → `predictBase(2026) = 0`。

**影响**：仅被直接调用 `predictBase` 的路径受影响。**引擎实际用 `getBase()`**，会向前回退到 2025 基数(8462)按预发年处理——实测 2026 四川场景 `getBase` 取 8631.24（8462×1.02 外推），不崩。故低优先。

**计划改造**（可选）：给 `PROV_BASE` 补 2026 键或修 `predictBase` 外推分支；或直接删除省份自带的 `predictBase` 让引擎统一走 `getBase`。

## ④ 广东 `PROV_BASE` 早期段占位 / `AVG_SALARY_HISTORY[2024]` 错值（待31省收尾清理）

- `guangdong.js` `PROV_BASE` 1978–2021 为外推占位值（注释 TODO：仅2022–2025有官方计发基数），2012 处存在断崖异常(5182)；但引擎算缴费指数优先用 `AVG`，早期 `PROV_BASE` 不参与实际计算，6案例全绿。
- `AVG_SALARY_HISTORY[2024] = 7646`，但用户提供数据源（2026-06-18 四川省历年平均工资表）2024 全口径 = **7894**（预估+5%）。7646 为错值，影响真实用户按裸工资算指数的精度（cases 注入 avg_index，未触发）。建议对齐到 7894。
