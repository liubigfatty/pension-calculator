## 二、2.0.3 包含哪些改动（相对 2.0.2 共 9 个相关 commit）

### 💳 支付链路（解 banned 根因）
- 虚拟支付完善：由标准 `wx.requestPayment` 全面迁移到 `wx.requestVirtualPayment`（道具直购模式）——虚拟商品走标准支付会被风控拦截，这是此前安卓 banned 的根因。
- `createOrder` 生成虚拟支付签名（offerId / env / goodsPrice 等 signData 字段类型与顺序已按官方文档固化校验）。
- `payCallback` 适配虚拟支付发货确认（`xpay_goods_deliver_notify` 推送 + 客户端支付成功后 best-effort 记录）。

### 🧮 引擎与计算
- B2 弹性提前退休路径修复（`getRetireTotalMonthsFlex` / `getDelayMonths` 的 `ef50` / `fw50` 分支缺失）。
- 记账利率、计发月数等边界修复。

### 📊 数据校正（**直接影响计算金额**，重点）
- 江西 2020–2026：计发基数改用官方「基本养老金月计发基数」，社平改用全口径城镇单位就业人员月平均工资（修正此前非私营口径错误）。
- 甘肃 1995–2025：社平 / 计发基数按甘肃省人社厅、税务局官方数据校正。
- **31 省配置副本全量对齐**：计发基数 / 社平工资 / 记账利率三字段在 6 个副本位置（provinces、cloudfunctions/calculate/provinces、data/provinces、js、docs/js 等）数据指纹完全一致，消除"校验通过但线上算错"的隐患。

### 🖼 报告与界面
- 报告页分享图改用原生 Canvas 2D（1:1 还原结果页视觉）。
- 报告屏显高亮、正常起领年龄显式标注、step2 / 审计留存修复。

## 三、更新流程（微信开发者工具 + MP 后台，共 7 步）

### 第 1 步：拉取最新代码并打开项目
- 当前已在 `main`，工作区干净（无需额外拉取）。
- 微信开发者工具 → 导入项目 → 选择本仓库根目录。
- 注意项目根识别：`miniprogramRoot = miniprogram/`，`cloudfunctionRoot = cloudfunctions/`（云开发项目）。

### 第 2 步：上传三个云函数（**关键，不只 calculate**）
在「云函数」侧边栏逐个右键 → **上传并部署（云端安装依赖）**：
- `cloudfunctions/calculate` ✅（用户已重传，但本次 2.0.3 仍有数据同步改动，建议再传一次确保一致）
- `cloudfunctions/createOrder` ⚠️（改了虚拟支付签名逻辑，**必须重传**）
- `cloudfunctions/payCallback` ⚠️（改了虚拟支付回调，**必须重传**）

> ⚠️ 历史坑：用户此前只重传了 `calculate`。2.0.3 的虚拟支付修复涉及 **createOrder + payCallback**，漏传会导致支付链路仍走旧逻辑。

### 第 3 步：配置云函数环境变量（虚拟支付能否跑通的关键）
1. 打开**云开发控制台** → 云函数 → `createOrder` → **配置** → **环境变量**。
2. 新增以下三项（值在微信支付/虚拟支付后台获取）：
   | 变量名 | 含义 | 来源 |
   |--------|------|------|
   | `VP_APP_KEY` | 现网 AppKey（敏感） | 虚拟支付后台「现网 AppKey」 |
   | `VP_APP_SECRET` | 小程序密钥（敏感） | 小程序 AppSecret / 虚拟支付后台 |
   | `VP_APP_KEY_SANDBOX` | 沙箱 AppKey（敏感，仅沙箱测试用） | 虚拟支付后台「沙箱 AppKey」 |
3. `payCallback` **不需要**环境变量。
4. `createOrder` 内已硬编码：`OFFER_ID='1450568626'`、`ENV=0`（现网）。如先做沙箱联调，需临时把 `CONFIG.ENV` 改 `1` 并配沙箱 key。

> ⚠️ 不配置这三项 → 支付时报"签名参数缺失，请确认云函数 createOrder 已重新上传（虚拟支付版本）并配置了 VP_APP_SECRET 环境变量"。

### 第 4 步：真机联调验证（上线前必做）
- **iOS + Android 各一台**，实测完整链路：**生成订单 → 微信支付 → 支付成功 → 报告解锁并可保存相册**。
- 重点验证：此前 `requestPayment:fail banned` 的安卓机型，本次应能正常拉起虚拟支付。
- 如先用沙箱：`createOrder` 里 `CONFIG.ENV=1` + 配 `VP_APP_KEY_SANDBOX`，沙箱支付用测试金额。

### 第 5 步：上传小程序版本 2.0.3
- 开发者工具右上角 **「上传」** → 版本号填 **`2.0.3`** → 项目备注填本次说明 → 上传。
- 版本号在上传对话框**手填**（仓库无版本字段）；已建 `git tag v2.0.3` 对应本次 commit 便于溯源。

### 第 6 步：提交审核
- 登录 [mp.weixin.qq.com](https://mp.weixin.qq.com) → **管理 → 版本管理 → 开发版本/体验版 → 提交审核**。
- 类目按实际填写（如 工具→计算器）。虚拟支付需在「付费能力」已开通（OfferID=1450568626）。
- 审核备注建议写明：本次修复虚拟支付通道（由标准支付迁移到道具直购模式，解决安卓 banned），数据按各省市人社厅/税务局官方口径校正。

### 第 7 步：审核通过 → 发布
- 审核通过后点 **「发布」**，可选全量发布或分阶段发布。

- **小程序端**：MP 后台「版本管理」可回退到 2.0.2 线上版。

## 六、注意事项 / 已知坑
1. 虚拟商品**必须**走 `wx.requestVirtualPayment`，标准 `wx.requestPayment` 卖虚拟商品会被风控 `banned`。
2. `goodsPrice` 单位=**分**：后台道具「价格(元)」=1 → 传 `100`；`offerId=1450568626`；`productId='pension_report'`。
3. 环境变量**切勿**明文写进代码或 commit（已在 `createOrder` 用 `process.env` 读取）。
4. 31 省数据以 `provinces/*.json` 为校验基准、云函数 `.js` 为线上生效位；本次已全量对齐，但发布后建议用真机抽测 2–3 个省（含双基数省广东/河南/辽宁、吉林）核对金额。
