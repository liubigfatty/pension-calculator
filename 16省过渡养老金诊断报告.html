官网重庆口径（渝办发〔2006〕205号）=「建立个人账户前缴费年限」= pre_account（含视同+建账前实际）。引擎按此算：case5✅、case4✅；case1/2/3/fc 的 expected 是脏值（case1 用 sight 漏 2.83 年、case2/3 的 sight=0 直接填 0）。同省4个案例，expected 混用 sight / pre / 0占位三种口径——基准本身就是乱的。

推论：cases 库 = 官方核定值 + 早期「verify --update 把引擎输出写回 expected」的回填值混合体（git 提交 1269a37 佐证）。部分案例后来重标定成官方值，但重标定不彻底，留下口径不一的脏数据。 我上一轮把重庆引擎改成「用 sight」只是凑巧让 case1 过了、却把 case5 搞错——这正是「用脏基准指导改引擎」的典型陷阱。

四、根因定论

  • 引擎侧：多数省过渡系数确实偏低/缺失（见上表），且广东/四川/浙江/重庆等省有专用公式分支，年限字段与指数口径需逐省核对官网。
  • 基准侧（更关键）：cases 库的 expected 口径不统一、含占位脏值，不能直接当"官方真值"去对齐引擎。否则改一个案例对、改乱另一个。

五、下一步方案（请你拍板）

A. 重建干净基准（推荐）：以官网官方口径为准，用引擎（或独立公式）批量重算全部171例的 expected，生成统一口径的干净基准，再逐省把引擎系数/公式对齐到新基准。彻底，但要改写 cases 的 expected 数据。
B. 先挑脏数据修正：只修正明显错误的 cases（sight=0 却填 0 占位、同省混用 sight/pre 的），再对齐引擎。范围小，但仍要改 cases。
C. 你给权威原表：你提供各省官方核定表的原始数据（像黑龙江那张表），我重新录入 expected，避免"重算"带来的二次误差。
无论选哪条，引擎系数都已按官网查到的正确值保留（重庆已改 1.4%、其余省缺/错的系数我列在表一，待你确认后统一写入）。当前 cases 全量：112 过 / 60 败。
